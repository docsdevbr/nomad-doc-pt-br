# Glossário

Este glossário fornece definições e explicações para termos e conceitos
importantes usados no Nomad.

## Allocation

Uma allocation é um mapeamento entre um task group em um job e um node cliente.
Um único job pode ter centenas ou milhares de task groups, o que significa que
um número equivalente de allocations deve existir para mapear o trabalho para as
máquinas clientes.
As allocations são criadas pelos servidores Nomad como parte das decisões de
agendamento tomadas durante uma evaluation.

## Authoritative and non-authoritative regions

A authoritative region é a região em um cluster multirregional federado que
contém a fonte da verdade para entidades replicadas entre regiões, como tokens
de ACL, policies e roles, namespaces e node pools.

Todas as outras regiões são consideradas non-authoritative regions e replicam
essas entidades extraindo-as da authoritative region.

## Bin packing

Bin packing é o processo de preencher recipientes (bins) com itens de forma a
maximizar a utilização dos recipientes.
Isso se estende ao Nomad, onde os clientes são bins e os itens são task groups.
O Nomad otimiza recursos ao fazer bin packing eficiente de tasks em máquinas
clientes.

## Cliente

Um cliente Nomad é um agente configurado para executar e gerenciar tasks usando
os recursos de computação disponíveis em uma máquina.
O agente é responsável por se registrar nos servidores, monitorar qualquer
trabalho a ser atribuído e executar tasks.
O agente Nomad é um processo de longa duração que faz interface com os
servidores.

## Data centers

O Nomad modela um data center como um agrupamento abstrato de clientes dentro de
uma região.
Os clientes Nomad não precisam estar no mesmo data center que os servidores aos
quais estão associados, mas precisam estar na mesma região.
Os data centers fornecem uma maneira de expressar tolerância a falhas entre
jobs, bem como isolamento da infraestrutura.

## Deployment

Deployments são o mecanismo pelo qual o Nomad implementa alterações graduais no
estado do cluster.
Deployments estão disponíveis apenas para jobs com o tipo `service`.
Quando uma evaluation é processada, o agendador cria apenas o número de
allocations permitidas pelo bloco [`update`][] e pelo estado atual do cluster.
O deployment é usado para monitorar a saúde dessas allocations e emitir uma nova
evaluation para a próxima etapa da atualização.

## Driver

Um driver representa o meio básico de executar suas **tasks**.
Exemplos de drivers incluem Docker, QEMU, Java e binários estáticos.

## Evaluation

Evaluations são o mecanismo pelo qual o Nomad toma decisões de agendamento.
Quando o _estado desejado_ (jobs) ou o _estado real_ (clientes) muda, o Nomad
cria uma nova evaluation para determinar se alguma ação deve ser tomada.
Uma evaluation pode resultar em alterações em allocations, se necessário.

## Job

Um job é uma especificação fornecida pelas pessoas usuárias que declara uma
carga de trabalho para o Nomad.
Um job é uma forma de _estado desejado_; a pessoa usuária está expressando que o
job deve estar em execução, mas não onde deve ser executado.
A responsabilidade do Nomad é garantir que o _estado real_ corresponda ao estado
desejado pela pessoa usuária.
Um job é composto por um ou mais task groups.

## Node

Um termo mais genérico usado para se referir a máquinas que executam agentes
Nomad no modo client.
Apesar de serem conceitos diferentes, você pode encontrar `node` sendo usado
de forma intercambiável com [`client`](#cliente) em alguns materiais e conteúdos
informais.

## Node Pool

Node pools são usados para agrupar [nodes](#node) e podem ser usados para
restringir quais [jobs](#job) podem criar [allocations](#allocation) em um
determinado conjunto de nodes.
Exemplos de casos de uso para node pools incluem a segmentação de nodes por
ambiente (desenvolvimento, staging, produção), por departamento (engenharia,
finanças, suporte) ou por funcionalidade (bancos de dados, proxy de entrada,
aplicações).

## Regions

O Nomad modela a infraestrutura como regiões e data centers.
Uma região conterá um ou mais data centers.
Um conjunto de servidores unidos representará uma única região.
Os servidores se federam entre regiões para tornar o Nomad globalmente ciente.

Em clusters federados, uma das regiões deve ser definida como a
[authoritative region](#authoritative-and-non-authoritative-regions).

## Server

Os servidores Nomad são o cérebro do cluster.
Há um cluster de servidores por região e eles gerenciam todos os jobs e
clientes, executam evaluations e criam task allocations.
Os servidores replicam dados entre si e realizam a eleição de líderes para
garantir alta disponibilidade.
Mais informações sobre os requisitos de latência para servidores podem ser
encontradas em
[Topologia de Rede](/nomad/docs/deploy/production/requirements#network-topology).

## Task

Uma task é a menor unidade de trabalho no Nomad.
As tasks são executadas por drivers, o que permite que o Nomad seja flexível nos
tipos de tarefas que suporta.
As tarefas especificam seu driver, configuração para o driver, restrições e
recursos necessários.

## Task Group

Um task group é um conjunto de tasks que devem ser executadas juntas.
Por exemplo, um servidor web pode exigir que um coprocesso de envio de logs
esteja sempre em execução.
Um task group é a unidade de agendamento, o que significa que todo o grupo deve
ser executado no mesmo nó cliente e não pode ser dividido.
